# 科研辅助

先确认输入文件、研究问题和期望产物；信息不足时明确缺口，不猜测论文内容、引用信息或实验结果。根据任务选择下列流程，可组合执行。

## 论文速读

1. 用 `glob` 定位 PDF、Markdown 或文本文件；用 `read` 读取文本。若只有 PDF，先用 `bash` 调用环境中已有的 PDF 文本提取工具，再读取提取结果。
2. 按“研究问题、核心贡献、方法、数据与实验、主要结论、局限性”提取证据；为关键结论保留页码、章节或原文位置。
3. 区分作者明确陈述与自己的推断；无法从材料确认的内容标为“未说明”。
4. 输出结构化速读报告，并给出值得复查的图表、公式或实验设置。

## 文献整理

1. 用 `glob` 收集文献文件，用 `read` 提取题名、作者、年份、研究问题、方法、数据集、指标、结论和局限。
2. 统一术语与字段；缺失字段写“未说明”，不补造 DOI、引用量或实验数值。
3. 按用户的研究问题建立对比表，再归纳共识、分歧、研究空白和可延伸方向。
4. 需要落盘时，用 `write` 生成 Markdown 报告；写入前先确认输出路径不会覆盖重要文件。

## 实验代码审计

在不真正运行完整实验的前提下，审查实验代码的可复现性、正确性和完整性。四个阶段：定位 → 分析 → 诊断 → 建议。

### 1. 定位：发现关键文件

1. 用 `glob` 扫描 `*.py`、`*.yaml`、`*.json`、`*.sh`、`requirements*.txt`、`environment*.yml`，列出项目文件结构。
2. 用 `grep` 搜索入口标记（`argparse`、`click`、`hydra`、`if __name__`、`main()`），确定训练/评估/推理的入口脚本。
3. 用 `read` 读取入口脚本的 main 段和参数定义段，提取所有可配置参数、默认值和类型。

### 2. 分析：检查可复现性

逐项检查以下维度，每项给出"✅ 已满足 / ⚠️ 有缺陷 / ❌ 缺失"三级判定：

| 检查项 | grep 搜索 pattern | 判定标准 |
|--------|-------------------|----------|
| 随机种子 | `seed\|random\|deterministic` | 所有随机源（Python random、numpy、torch、cudnn）均已固定，且种子值可通过参数配置 |
| 数据划分 | `train_test_split\|kfold\|shuffle\|split` | 划分策略明确，随机种子传入划分函数，划分结果可复现 |
| 依赖声明 | 直接 `read` `requirements.txt` 或 `environment.yml` | 所有第三方包均有版本约束（`>=` 或 `==`），无未声明的 import |
| 硬编码路径 | `/home/\|/data/\|C:\\\\\|/mnt/` | 无绝对路径；数据/输出路径均通过命令行参数或配置文件传入 |
| 评估指标 | `accuracy\|f1\|bleu\|rouge\|precision\|recall\|mse\|mae` | 指标计算在单卡/单 batch 上的正确性（非 batch 内误平均），与论文声明一致 |
| GPU/资源 | `cuda\|gpu\|device\|batch_size\|num_workers` | 设备选择可配置（`--device` 或自动检测），batch_size 可调，无硬编码 `cuda:0` |

对判定为 ⚠️ 或 ❌ 的项，记录具体文件、行号和缺陷描述。

### 3. 诊断：环境快速验证

用 `bash` 执行**仅**毫秒级的安全检查，不涉及训练循环：

```bash
python --version                    # Python 版本
python -c "import torch; print(torch.__version__)"  # 关键包是否存在
python train.py --help 2>&1 | head -30  # 参数列表（若脚本支持 --help）
python -c "import train" 2>&1       # 语法检查（不执行 main）
```

失败时区分两类错误并给出修复建议：
- **环境错误**（ModuleNotFoundError、版本不兼容）→ 建议安装/升级命令
- **代码错误**（SyntaxError、NameError）→ 指出错误位置和修复方案

### 4. 建议：输出审计报告

用 `write` 生成结构化审计报告，按以下格式：

```markdown
# 实验代码审计报告：<项目名/论文名>

## 项目概况
- 入口脚本：列出所有入口文件路径
- 关键参数：列出影响实验结果的核心参数及其默认值
- 依赖规模：包数量，Python 版本要求

## 可复现性检查

| 维度 | 判定 | 说明 |
|------|------|------|
| 随机种子 | ✅/⚠️/❌ | 文件:行号，具体问题 |
| 数据划分 | ... | ... |
| 依赖声明 | ... | ... |
| 硬编码路径 | ... | ... |
| 评估指标 | ... | ... |
| GPU/资源 | ... | ... |

## 严重问题（❌ 项）

逐条详细说明：
- 位置、缺陷描述、复现影响、修复建议（含具体代码示例）

## 环境验证结果

- 命令、退出码、输出摘要
- 失败项的分析与修复命令

## 总结

- 可复现性评分：N/6 项通过
- 建议在复现前修复：列出 ❌ 和 ⚠️ 项
- 预估资源需求：从代码中推断的 GPU 显存、数据大小、预计运行时间（若有说明）
```

### 注意事项

- **不运行完整实验**：bash 只用于 `--help`、`import` 检查、`pip list` 等毫秒级命令。不执行训练、推理或数据处理脚本。
- **不安装包**：发现缺失依赖时列出安装命令，但不自动执行 `pip install`。
- **不猜测路径**：未找到配置文件或数据路径时标注"未发现"而非臆测。
- 环境验证失败时，区分"脚本 bug"和"本机环境差异"——前者是真正的问题，后者可能只是本机缺少某些包。
- 报告中引用结论时附文件路径和行号。

## 注意事项

- 论文解析失败时说明受限原因，并请求文本版或可复制内容。
- 审计实验代码时，bash 仅用于毫秒级安全命令（`--help`、`import` 检查、版本查询）；不执行训练、推理或数据处理脚本，不自动安装包。
- 运行未知脚本前先检查其参数与副作用；不执行删除、上传、安装或长时间高资源任务，除非用户明确授权。
- 引用结论时附可追溯位置；报告中的推断使用”可能””表明”等措辞。

