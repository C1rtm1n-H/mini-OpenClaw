# 实验错误诊断

## 流程

1. 用 `read` 读取完整错误信息，用 `grep` 定位最后一个异常及其 `Caused by`、文件和行号；不要把后续连锁错误当作根因。
2. 用 `glob`、`grep`、`read` 检查相关源码、配置、依赖声明和输入路径，建立“症状 → 首个失败点 → 根因”的证据链。
3. 将根因分类为环境、依赖、数据、路径、资源、配置或代码逻辑；证据不足时标为“疑似”，不要臆测。
4. 给出影响范围最小的修复方案。未经用户授权，不安装依赖、不下载数据、不修改代码、不启动完整实验。
5. 优先执行只读检查；确需命令验证时，只运行低成本、无训练、无联网、无破坏性的命令，并记录命令、退出码和输出摘要。
6. 输出根因、证据位置、最小修复、验证结果和仍未确认的事项。

## 常见检查

- `ModuleNotFoundError`：对照 import 与依赖文件，区分未安装和未声明。
- CUDA/OOM：检查 device、batch size、精度和显存信息，不估造所需显存。
- shape/dtype/device mismatch：追踪张量来源和首次不一致位置。
- 路径/编码错误：检查路径来源、平台差异和实际编码。
- checkpoint 错误：检查模型结构、键名、保存版本和加载参数。

## 输出格式

```markdown
# 实验错误诊断
- 症状：
- 首个失败点：文件:行号
- 根因：已确认/疑似
- 证据：
- 最小修复：
- 验证：命令、退出码、摘要
- 未确认事项：
```
